# Failure-mode catalog — conquering-campaign

Full prose for every numbered failure mode. SKILL.md carries the compressed cross-reference table (`#N | title | status | cure pointer`); this file carries the diagnostic narrative + post-mortem context. Look up `#N` here when you need the *why* and the *how-it-was-found*, not just the cure pointer.

**Status legend:** `promoted → PN` (now a Standing Principle) · `named → X` (superseded by a named pattern) · `embedded → §X` (cure lives in that SKILL.md section) · `active` (watch-list trap; cure section + this prose).

Adding a new mode: append here AND add a row to the SKILL.md table. Next sequential `#N`. Cross-refs stay stable.

1. **Skipping W0 when Ollama is available.** *Standing principle P1.* Free tokens wasted. Every Claude subagent re-explores from scratch instead of compressing against the pre-scan.
2. **Missing model assignment table in the plan.** *Standing principle P2.* No audit trail for which model did what. Token spend is unaccountable.
3. **Stopping mid-execution to ask the user.** *Standing principle P3.* The planning interrogation exists to prevent this. If Claude is stopping, the interrogation was incomplete — note it for next time.
4. **Inventing a new style instead of following the existing one.** *Standing principle P4 + P9.* The app already has a design language. Claude builds a page that looks different from the other 26 admin pages. The conformity discipline + reuse-before-build gate (G1) exist to prevent this.
5. **Copy-pasting patterns instead of extracting.** *Standing principle P5.* The campaign creates more duplication than it resolves. The consolidation discipline exists to prevent this.
6. **Forgetting dead-code cleanup.** Legacy components left behind after migration. The cleanup plan in Step 2 and the deletion step in Step 5 exist to prevent this.
7. **Skipping the vault-log.** *Standing principle P6 + P10.* The log must be written immediately at Step 5, not "when the user asks." No "small tweak" or "same conversation" exemption.
8. **Audit mode — Trusting a graph hypothesis without code verification.** Always verify before recommending a refactor. A graph "god-object" label needs LOC + public-export count confirmation, not edge-count alone (high fan-in ≠ god object).
9. **Audit mode — Skipping the synthesis step.** Without `99-synthesis.md`, the doc rewrites drift from the evidence.
10. **Build mode — Running `apply_migration` without approval on a red-gated phase.** Even under `autonomous` cadence, red phases always gate.
11. **Build mode — Skipping the risk sweep.** `99-risk-sweep.md` catches "we fixed Insights but broke Notifications because they share a trigger." The artifact must EXIST on disk before `status: completed` — two 2026-05-29 campaigns (`saved-views-app-wide`, `db-overhaul-v2`) flipped completed with no risk-sweep file. The §5.5 status flip is gated on the file existing.
12. **Both modes — Leaving the campaign plan with `status: in-progress` after finishing.** Update it to `completed`.
13. **Both modes — Stalling after the plan.** If the plan is approved, execution begins immediately. The 37% stall rate in past campaigns happened because this wasn't enforced. *Extension:* stalling AFTER close (vault-log done, version bumped) waiting for commit permission is the same trap — when the user granted standing perms ("you have all necessary perms to finish the job"), the commit is part of close, not a separate ask.
14. **Pre-committing deletions in the plan.** The plan promises "delete X, Y, Z" before any consumer audit. Reality: most candidates are still load-bearing after the migration — chrome moved but the inner content stayed (legacy filter bars, summary bars, topbars often survive a chrome migration intact). The plan must list **candidate** deletions, the cleanup phase must grep-prove orphan status before any `rm`, and the change-log must honestly record what was kept vs deleted. Distinguish two kinds of dead code: (A) deletable after *chrome* migration (only call site is the chrome layer being replaced); (B) deletable only after *content* migration (still used as inner content). Type B cannot be pre-committed in a chrome campaign — `portal-ui-unification` planned 9 deletions, only 1 was actually deletable.
15. **Mass-rename without a gate.** Standing "don't pause until done" instructions don't cover architectural one-way doors. A 25-file namespace move is reversible only at the cost of a churn cycle. Any phase that mass-renames imports across >15 sites, moves a top-level namespace, or alters a public component name must hit a single-question `AskUserQuestion` gate first — even under `autonomous` cadence. 30 seconds of approval beats hours of rollback. *Note on cost-reasoning:* "avoid churn, keep the old namespace" is usually wrong — 33 targeted import renames are cheaper than carrying a misleading namespace forever. Prefer the accurate name at extraction time.
16. **Ollama prescan ANSI/thinking-trace pollution.** Piping `ollama run` straight to `tee`/file captures spinner frames and `Thinking…/…done thinking` lines from chat-tuned models. The artifact is unreadable. Always pipe through the sed/grep filter in §0.5. *Sharper variant (rule-not-working in practice):* the prescan may run "successfully" yet contain hallucinated paths or echo-back garbage — the main agent must spot-check 3–5 prescan paths against real Read/grep and record `w0_useful: yes|no|partial` BEFORE dispatching any W1 subagent. A corrupt prescan shipped to W1 wastes every W1 token (`portal-ui-unification` shipped echo-back garbage; main agent had to rewrite the prescan from verified reads).
17. **Preview-verification thrash on every Edit.** PostToolUse hooks may remind you to preview-verify after every observable edit. During an in-flight mass refactor the build is intentionally broken between Edits; preview verification then is meaningless. Verify at **phase exit only**. *Single-turn cosmetic phases:* when one Edit completes the whole phase, the turn IS the phase — verify once at the turn's end. *Critical override (see P8 / #B-conflict):* when the project's standing instruction / memory says preview verification is the USER's job (lex_council: `[[feedback_no-dev-server-hijack]]`), the agent does NOT run preview tools at all — it hands off a verification recipe for the user to run, and NEVER starts/kills the dev server.
18. **Build toolchain cache (Turbo) masking pre-existing errors.** `npx turbo run check-types` returns a cached "pass" when Turbo's task hash hasn't changed — even after file edits, if those files are not in Turbo's tracked input set. On the *first* verification run per phase, always add `--force`. When the forced run surfaces errors in files you never touched, confirm with `git log -- <file>` they predate the campaign — document as `CHK-<id>` and continue. Never treat a cached "pass" as ground truth; never attribute pre-existing errors to the campaign. *Read-back corollary:* after editing a shared token sub-object, the Edit tool can report success while the bytes never hit disk (cache illusion) — re-Read the file to confirm + `--force` tsc on the first consumer of any new token.
19. **UI consolidation: matching visual hierarchy requires reading the full render function.** The first draft of a new row/card looks structurally wrong (flat table-like columns) even with correct tokens — because it was modelled from the old component's structure, not from reading the reference's full render JSX. Import lists, prop types, and signatures don't reveal layout hierarchy. Read the full render function; extract named line-levels explicitly. *Promoted to §Step 3 W3 step 1.5a + #26.*
20. **`tsc + lint` green = false-confidence trap for client-state features.** For UI / theme / auth / hydration / realtime / locale / i18n campaigns, compile-time checks confirm the code parses — they cannot detect a broken user-facing outcome. The dark-mode rollout v1.0 shipped P0–P4 green while the visible feature was broken on every non-admin page (SSR-hydration mismatch + FOUC). Browser verification at phase exit is mandatory for these surfaces. The surface-coverage matrix needs evidence (screenshot, polled DOM read, console log, network call) behind every "✓" — not intent.
21. **Diagnostic state pollution.** Manual pokes during debugging (`setAttribute`, `localStorage.setItem`, `setState`) persist into subsequent verification reads in the same page session. Without a `location.reload()` between "set state to debug" and "read state to verify," the next read sees your mutation, not the system's behavior — false "it works!" conclusions that evaporate on a real load. Treat verification reads as cold-cache snapshots; reload between any manual poke and the next observation.
22. **Design-language matching from the wrong or stale reference.** When told "match X's button style," the style lives in X's action-bar or leaf primitive, not at the page-level row. `DocListRow`→`DocActionBar`→`TableActionBtn` (solid navy); `TaskListRow`→`TaskListActionBar`→ghost buttons — two hops down. Reading only the row gives the wrong picture; follow the import chain to the leaf. The reference may have been edited this session — re-read immediately before implementing. *Strengthened (2026-05-29 mining): name the EXACT leaf file path in the plan and extract every token (background, border thickness+token, borderRadius token-not-px, icon size, icon color token, transition, wrapper, hover mechanism) before writing — approximating one token (`8` instead of `borderRadius.lg`, `title=` instead of `<Tooltip>`) causes serial revert round-trips (`c013271e`: 3 reverts before the icon buttons matched).*
23. **Cascading-default leak.** A phase changes a shared primitive's visual default and stops at the primitive's file. Every consumer that passed an explicit override against the OLD default is now silently broken (invisible icon on a new transparent bg). Surfaces as N follow-up one-shot turns that compound into hours. The §W3 stale-override sweep is the gate; pair with §Consolidation rule #4 (grep before changing a shared default).
24. **Post-closure tweak drag.** The campaign closes; the user then fires N short tweaks over days that touch the same surface, each handled as a fresh one-shot turn, bypassing the sibling sweep + conformity audit + re-anchor + stale-override sweep. §5.8 classifies each as stub-extension (log against predecessor) or mini-extension (re-open as EXTENSION). "**fix X too**" signals the predecessor's re-anchor missed an adjective.
25. **Same-session mid-flight pivot misclassified as a new campaign.** User re-invokes `/conquering-campaign` minutes after close, pointing at a deficit. Falls between §0.0 (only `in-progress`), §Step 4 (only W4), §5.8 (only days later). §0.0.1 is the cure: skip W0, append a phase with `discovered_during: same-session-pivot`, edit the vault log in place.
26. **Plan named the reference but never extracted its render structure.** Plan says "model X on Y" and stops. W3 reads Y's imports/types/signature — none reveal that Y renders `TableShell + flex column + FileListRow` cards, not a `<table>`. §Step 3 W3 step 1.5a makes the `## Reference render structure` block a required plan section + pre-execution gate.
27. **Nth hand-rolled clone of a recurring shape.** The campaign creates a 3rd `*ListRow.tsx` mirroring two siblings line-by-line without composing a base primitive. The Consolidation check fires at plan time against whatever inventory was listed; if the inventory missed the base primitive, it never triggers. §Step 3 W3 step 1.5b is the cure: a write-time grep counting sibling implementations BEFORE the Write, plus a re-read of the docs for the expected base component name.
28. **Multi-Edit atomicity trap.** `Edit #1` adds a helper; `Edit #2` adds its call site. The PostToolUse linter fires *between* Edits, sees the helper as `no-unused-vars`, strips it; `Edit #2` then fails or breaks. When a phase needs ≥2 sequential Edits where the intermediate state would fail lint, use a single `Write` over the whole file — atomic, linter sees only the final state.
29. **TodoWrite list survives across same-session campaigns.** Stale `[completed] Phase 2` items from an earlier campaign nag mid-flight and mislead the next invocation. §0.0.2 mandates a `TodoWrite` reset before W0/W1 of a fresh campaign.
30. **W0 prescan ships echo-back garbage and downstream agents cite it as ground truth.** Output mirrors the prompt's headers without inserting file insight (`<file> — <line count>` instead of real counts). §0.5.1 quality gate: read the prescan against a concrete-signal checklist, mark `w0_useful: no` when echoed, instruct W1 to read files cold. Also skip W0 for builds <10 files.
31. **Standing instructions silently inherited across same-session campaigns.** "work without stopping" set in campaign 1 propagates silently through 2/3/4; the new plan never records it. §1.2 Q13 + `standing_instructions:` frontmatter force quoting the verbatim instruction + naming the predecessor. *Conflict corollary:* an inherited instruction that CONTRADICTS the new campaign's goal (e.g. "keep everything old" inherited into a cleanup campaign) must be explicitly lifted + replaced in the plan, not silently carried.
32. **W1 subagent mis-locates a DB column and the migration inherits the error.** Subagent claims `is_programmer` lives on `council_members`; it's actually on `admin_perms`. Patterns: tables renamed mid-history, columns on overlay tables, columns exposed via a join view. §Step 3 W3 step 1.5d: `execute_sql` against `information_schema` / `pg_policies` / `pg_views` before writing the migration body.
33. **`apply_migration: success` trusted as ground truth after a prior failed retry.** `CREATE OR REPLACE VIEW` silently drops `security_invoker`; RLS can apply a different qual than the body; triggers can fail silently without `set search_path = ''`. §Step 3 W3 step 2a: post-migration ground-truth SELECT recorded as evidence. *Concurrent-actor extension:* production schema can flip mid-session from another actor's migrations — a subagent's live read can be stale before it executes (`v2-trigger-column-drift` drafted nickname fixes against a snapshot a parallel campaign had already reversed; the fix would have hard-broken prod). Re-probe live schema + check the `supabase_migrations` ledger at the START of every wave, not just at plan time; STOP + reconcile if interleaved foreign migrations appear.
34. **Phase plan tries to inline >5KB of generated content.** A 75KB locale JSON overflows the planning subagent's response, truncates, or hallucinates middle sections. §Step 2 `generation_strategy: deferred-parallel-subagents`: the plan stores the spec (path, source locale, key count, sampling rule); W3 spawns per-file subagents that write to disk; the change-log records sizes + a diff sample as evidence.
35. **"Page renders + no console errors" skips the campaign's actual value (CRUD/editor/multi-step).** The Save button was never clicked; persistence-across-reload never ran. §Step 1.2 Q9 requires BOTH a render recipe AND an interaction recipe; §Step 2 `verification_class: interactive` phases can't be verified by render alone. The interaction recipe must include the round-trip (write → reload → confirm persisted → revert → reload → confirm restored).
36. **Pre-existing runtime errors attributed to the campaign or vice-versa.** `git diff <start-sha>..HEAD | grep -F '<error>'`: absent → pre-existing (`spawn_task` + continue); present → campaign-introduced (phase blocker). Provenance first, classification second, action third. *Working-tree corollary:* an abandoned earlier campaign's dirty edits survive into a new campaign and cause misattributed lint/CHK failures — check `git diff HEAD` for unrelated dirty files before blaming the new campaign.
37. **Same-session pivot-chain accretes scope past artifact recognisability.** By pivot 3+, close-out overhead dominates and the campaign drifts past its named shape (an "Actions tab" campaign ends with 2 DB tables + a visibility audit). §0.0.3: detect the chain (≥2 prior pivots + >6 phases + new surface), batch the close-out, present a single spin-off `AskUserQuestion`.
38. **Preview verification claimed "deferred" every phase under a permanent auth wall.** 30–60 wasted preview tool calls per campaign. §0.0.4: one probe at session start, set `preview_verification: blocked-by-auth`, stop adding preview steps. (Distinct from the P8 "preview is the user's job" override, which means hand-off, not deferral.)
39. **Long-running batch script with stdout buffering = opaque wait.** Node's stdout is line-buffered to a non-TTY; `tail -f` shows nothing until exit. Any batch looping N>20 items MUST write progress to a sidecar via `fs.appendFileSync('progress.log', …)`, not `console.log`. The main thread tails the sidecar; it also survives a kill, enabling resume. *See also #75 (no-heartbeat).*
40. **Pivoting strategy mid-batch discards N partial calls.** A batch writing its manifest only at end-of-run loses 80/92 completed summaries on a kill. (a) Checkpoint incrementally (every K=5 for ≤100, K=20 for >100); (b) one `AskUserQuestion` confirming strategy before any >5min batch.
41. **File mutated by background tooling between Read and Edit (long-pause variant).** Between a >30s wait (subagent / Monitor / MCP) the file is silently mutated by a linter/formatter/parallel agent; next Edit returns `File has been modified since read`. After any wait >30s, re-Read the target before the next Edit.
42. **Source-walk subagent output bypasses the JSON-on-disk pattern.** Inline returns force fragile regex parsing and vanish from the campaign folder. Every source-walk subagent writes full findings as JSON to `<campaign>/NN-<topic>.json` and returns only a short summary. *Named → source-walk template.*
43. **DB sub-pattern reinvented per phase instead of named.** Probe → draft → approve → apply → verify, re-described every phase. *Named → PDAAV* (see references/db-playbook.md).
44. **Symptom diagnosed from code, not from observation (bug-investigation variant of #20).** Vague symptom ("kinds shows nothing") → jump to code → fix one of N hypotheses → ship on green. §1.4: split into Part A (reproduce + confirm interpretation, one short question) and Part B (plan fixes). Cost of asking ~30s; cost of fixing the wrong cause is a full close-out + a redo campaign.
45. **Express-mode acknowledgment gap — full pipeline on small builds.** A 4-file fix gets 10+ artifacts of ceremony, so users stop invoking the skill for small things. §Step 2 `scope: small` declares explicit trims (skip W1/W2/per-phase files/risk-sweep) while keeping §1.4 + §1.5d + §2a + §2a-RLS + §5.3 + §5.4. *db-intensive small note:* a focused feasibility memo that runs deep MCP reads should declare `scope: small + db-intensive` so the express trim isn't read as "skip MCP."
46. **Vault-log + campaign-plan split across docs roots (monorepo wikilink breakage).** §0.0.5: probe both docs roots at session start, resolve `campaign_folder_root` + `vault_log_root`, co-locate by default, record both in frontmatter. *(This very skill hit defect #1 of a sibling shape — companion files written to a staging dir, never co-located with SKILL.md.)*
47. **RLS qual-text matches ≠ effective-access matches.** Policy text can match intent while a restrictive policy, a grant, or a `security_invoker=false` wrapping view blocks/admits the user. §Step 3 W3 step 2a-RLS: run a representative SELECT under a simulated authenticated context for an admit-user AND a deny-user, record counts. Bundle ≥2 access-control-layer changes as sub-phases (§2b), verify each layer independently.
48. **Dev-server module cache for newly-added JSON keys (runtime mirror of #18).** A key added to `messages/<locale>/<ns>.json` mid-session isn't picked up — the dynamic import was evaluated at startup. Runtime renders the raw key path. Detection trio: (a) raw key path renders, (b) `git blame` shows `00000000 (Not Committed Yet)`, (c) sibling keys render fine ⇒ restart the dev server, don't chase the DB/config. Umbrella: *if the runtime symptom contradicts the current file on disk, restart before diagnosing.*
49. **Sidebar nav primitive lock-in violation (specific case of #27).** A new sidebar card hand-rolls a `<button>` instead of the canonical nav-row primitive, shipping `background:<active>` + `color:<same-family>` = invisible label. Any `<button>` inside `<SidebarCard>` that's a selectable list item uses the canonical nav-row — no exceptions for "dev-only" panels.
50. **Single-property follow-up tweak shouldn't re-enter campaign machinery.** For a 4-line CSS tweak the plan file alone exceeds the patch. §Step 2 `scope: micro` (≤2 files, ≤10 lines, no new artifacts): no plan file (conversation IS the plan), no per-phase log; kept mandatory — §1.4 reproduction, tsc+lint, preview/hand-off, §5.3 vault-log (P8 non-negotiable).
51. **JSX structural pivot via chained Edits.** Wrapping N>3 JSX siblings with the same conditional via N chained Edits: hard-to-anchor, lint sees an inconsistent file, fragmented diff. Distinct from #28 (lint-dirty) — here the intermediate is lint-clean but structurally inconsistent. *N>3 same-shape wraps in one file ⇒ single `Write` (or one `replace_all` Edit).*
52. **Misnamed themable token reached for as if static.** `C.white = var(--color-secondary-bg)` is `#FFF` in light, `#282828` in dark; reaching for `C.white` as "white text" ships dark-on-dark in dark mode. §Step 3 W3 step 1.5e: themable-token semantic audit at W1 — grep the token file for static-named tokens bound to CSS vars, document safe replacements, grep + fix all misuse in one mass-edit phase. Use the `alwaysWhite` escape-hatch for true-white foregrounds.
53. **Visual-contrast fix shipped without WCAG measurement.** `alwaysWhite` (#FFF) on `C.gold` (#C9A84C) = ~2.8:1, FAIL AA. §Step 1.2 Q9 + `verification_class: contrast-measured`: measure the ratio (`getComputedStyle` + compute, or DevTools panel), record it, expect ≥4.5 (normal) / ≥3.0 (large/UI). Run ≥2 probe passes — the first finds obvious FAILs, the second finds asymmetric pairs.
54. **Express mode trimmed silently without declaring `scope: small`.** `phases_remaining` skips W1/W2/per-phase files but frontmatter lacks `scope: small` + its required fields → future readers can't tell deliberate trim from sloppy plan. §Step 2 declaration gate: before W3, if phases are trimmed, the scope tier + all required fields MUST be present.
55. **Stub-extension vault logs invisible in successor's predecessor chain.** `A (closed) → stub-fix (vault-log only) → B (EXTENSION)`; B's `predecessor:` points at A, skipping the stub-fix. §5.8: B's "Trigger / Context" lists every intermediate same-session vault log as a one-line wikilink.
56. **`git stash` provenance check meaningless for untracked files.** `git stash` skips untracked by default → post-stash check compares identical states. Pass `-u`, or use `turbo --force` + manual grep/inspection. The git-diff pattern also presumes the file was tracked at campaign-start.
57. **Directional-adjective migration verb infers the wrong keeper.** "migrate from files-vertical to files-horizontal" — the keeper file was named `FilesStackHorizontal` but the keeper render mode was `layout="vertical"`. §1.4.1: fire a single "which side is the keeper?" question BEFORE §1.2 locks. `Q1-inverts-scope ⇒ Q2..QN stale; re-batch.`
58. **N=3+ enum-prop union left as a stale 1-value union after consolidation.** Consolidation migrates consumers off dead union values but leaves the prop, the dead render branches, and helpers used only by them. §Consolidation rule 5: grep each union value's consumers; if N-1 cluster on one keeper, drop the prop + dead branches + dead-only helpers in the same phase.
59. **Express vs full mismatch on medium-scope builds (6-20 files).** Express too tight, full too heavy. §Step 2 `scope: medium`: 6-20 files + ≤1 migration + no >15-site rename + ≤1 forced gate; trims like small PLUS a standalone `99-risk-sweep.md`.
60. **Rename-with-shrink chained-Edit trap.** Rename a public component + delete >40% of its body via chained Edits → orphan helpers, type-errors, linter strips mid-flight. *Named → RIBS:* `git mv` → Read → single atomic `Write` → barrel update.
61. **Project-wide lint conflated with campaign-scope lint.** A pre-existing warning in an untouched file fails `--max-warnings 0`. §5.2 scoped-lint: after provenance confirms pre-existing, run `eslint` scoped to the campaign-touched file list (from `git status --short`); `EXIT=0` proves the campaign surface clean; the full-project warning rides as a `CHK-<id>` / `spawn_task`.
62. **Asymmetric-themable-color pair: no foreground passes AA in both modes.** Sibling of #52 but the binding is correct; the trap is hue-asymmetry across modes. `C.green` light = dark forest #155724, dark = bright Material #4CAF72 — NEITHER `alwaysWhite` nor a flat dark fg gives ≥4.5 in BOTH modes. Cure = the **bg-lock pattern**: for brand-saturated solid color buttons/badges needing a fixed fg, lock the BG via theme-flat hex matching the light value (`bg: '#1B6E4B' /* theme-flat */`) so `alwaysWhite` passes AA in both modes. Applies only to solid accent BGs, not canonical chip pairs (`successBg + successText`, tuned for both). Detection: in the post-grep contrast probe, run ratios in BOTH modes per pairing. §Step 3 W3 step 1.5e sub-discipline. Lex memory `[[feedback_green-bg-themable-asymmetry]]`.
63. **Pre-commit hook auto-commits between user-initiated git operations.** Hooks (husky / lefthook / simple-git-hooks, often i18n codemod packages) run their own `git add`+`commit` chain on each user git op; the user's commit lands with a different message, the tree dirties between `status` and `commit`, the submodule pointer advances autonomously. Detect via `cat .git/hooks/pre-commit .git/hooks/post-commit` + `git config --get core.hooksPath` + grep `package.json` BEFORE any commit. Don't chase the cascade — let it finish, commit final state once, discard hook-leftover dirty files via `git checkout -- <file>`, document the chain in the vault-log.

## Added 2026-05-29 (v3.0.0) — from the exhaustive 136-campaign / 159-transcript mine

64. **Z-index numeric fix is structurally wrong for popover clipping.** Every "dropdown behind widget" report had the same root cause — ancestor stacking-context containment, not the z-index number. Bumping the number never works permanently (20 ad-hoc values existed). Cure: `createPortal` the popover to `document.body` + fixed positioning from `getBoundingClientRect()`, and adopt a canonical Z token scale (`base:0, sticky:100, sidebar:200, topbar:300, overlay:8000, modal:9000, dropdown:9500, toast:9999`). All new popovers/dropdowns use `Z.dropdown` + `createPortal`. (`tasks-visual-bug-fixes`, z-index chaos campaign.) See references/fe-i18n-playbook.md.
65. **Sidebar `overflow:hidden` clips portal children.** Any absolutely-positioned child (dropdown, popover, picker) inside a scrollable/`overflow:hidden` container is clipped; a higher z-index does nothing because the clip is the container, not the stack. Cure: `createPortal` to `document.body`. Sibling of #64. (`tasks-visual-bug-fixes` W0 prescan flagged this as distinct from the z-index chaos.)
66. **Stepper / label-window collapse by step-count is provably insufficient.** Overflow = f(step_count, panel_width, label_length, locale). A 5-step modal at 640px with wide Arabic labels still overflows even though the collapse threshold was "› 5". The only correct fix is runtime measurement: `scrollWidth` vs `clientWidth` in a `ResizeObserver`, then collapse. Any new stepped modal uses the self-measuring primitive, never a hardcoded `labelWindow` threshold. (`form-popup-overflow-elimination`.)
67. **Postgres function bodies are TEXT, not OID-bound — renames don't auto-fix them.** `ALTER TABLE … RENAME` (column or table) auto-resolves views / RLS policies / triggers via OID, but NOT `pg_proc.prosrc` — function bodies reference columns by literal text. A missed body stays silent until something forces a re-plan (e.g. a `VOLATILE → STABLE` change altered planner eagerness and surfaced a broken column reference long after the rename). Cure: on ANY column/table rename, grep ALL function bodies (`pg_get_functiondef` across `pg_proc`, no language filter — SQL functions are missed by plpgsql-only scans) for the old name and rewrite each; a `DROP … ` without CASCADE is a useful pre-flight dependency check. (`name-columns-consolidation`; lex memory `[[discovery_sql-functions-missed-by-plpgsql-scan]]`.) Extends #33; see references/db-playbook.md.
68. **DB cascade audit is blind to FE callsites.** `pg_depend` maps DB-side consumers only (views, policies, triggers, functions). A `.from('tsl')` in `app/api/tasks/suspend/route.ts` and a `SettingsPanel.tsx` reading `users.nickname` directly both survived DB-side-clean DROP/rename phases and would have broken production. Before any DROP / rename, grep `.from('<table>')` and `.select('<col>')` / direct reads across `app/api/`, `lib/mutations/`, `hooks/`, and component files — not just the DB dependency graph. (`tasks-list-perf`, `nickname-to-user-preferences`.) See references/db-playbook.md.
69. **Stale planner statistics cause "query" timeouts that aren't query problems.** A 6,697-row table with `last_autoanalyze = NULL` had its row estimate frozen at 122 → the planner chose catastrophic nested-loop plans → statement timeout. `ANALYZE <table>` was the entire fix; no query rewrite, no index, no RLS change. Before diagnosing any slow/timeout query — and after any big data backfill — check `pg_stat_user_tables.last_autoanalyze` / `n_live_tup` vs reality and run `ANALYZE` first. (`docs-s-js-access-perf`.) See references/db-playbook.md.
70. **Trigger firing-order + AFTER-trigger self-UPDATE recursion.** (a) Postgres fires BEFORE row triggers in **alphabetical** order — adding a new BEFORE trigger alongside an existing one requires choosing the name to control order (`aa_users_access_btr` deliberately sorts after `aa_branches_access_invariant_btr`); document the intended order in the trigger COMMENT. (b) An AFTER trigger that `UPDATE`s its own table re-fires all AFTER triggers (including itself) → recursion / double-compute; the cure is a BEFORE trigger that mutates `NEW` directly instead of issuing an UPDATE. PDAAV doesn't currently warn about either. (`docs-s-js-access-perf`, `name-columns-consolidation`.) See references/db-playbook.md.

Surfaced in ≥2 campaigns. Always use the namespaced form; grep for `useTranslations()` no-arg calls during i18n campaigns. (`public-tier1-funnel-hygiene`, `i18n-hardcoded-sweep`.) See references/fe-i18n-playbook.md.
72. **i18n JSON hygiene cluster.** Four distinct silent-breakage traps when editing locale JSON: (a) deleting the alphabetically-last key in a block leaves a trailing comma on the prior line → webpack rejects the file (canonicalize the JSON after any key delete, don't just remove the line); (b) renaming an EN key silently orphans every DB `key_path` override pointing at the old path — next-intl ignores the orphan and renders the new default with no warning (cross-reference key_path against live JSON before saving; cron-flag orphans); (c) a translated string that reorders interpolation placeholders (`{month} {day}` → `{day} de {month}`) is only safe with NAMED params (`t('key', { month, day })`), never positional; (d) register a locale in `routing.ts` only AFTER its message files exist (reverse order = next-intl accepts the URL but throws/renders empty). Also re-run the dead-key grep after any phase that ADDS `t()` calls — a key purged in P1 can become live again in P2 (#cross-phase resurrection). See references/fe-i18n-playbook.md.
